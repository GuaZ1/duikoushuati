# AI 开发人员工作规范

> 本文件面向所有参与「刷题小程序」项目的 AI 开发人员。每次开始开发工作前，请完整阅读本章，并确保后续所有代码、配置与操作均符合以下规范。

## 1. 核心原则

- **以用户需求为唯一目标**：只做用户明确要求的事，不擅自添加功能、不提前做抽象、不写"未来可能用到"的代码。
- **可逆优先**：优先采取可撤销、影响范围小的本地操作；涉及删表、删分支、删文件、强制推送等破坏性操作必须先取得用户确认。
- **验证驱动**：每完成一个功能或修复，必须在本地运行并验证；无法验证的改动必须明确告知用户。
- **透明沟通**：遇到阻塞、歧义或需要用户决策的问题，立即暂停并询问，不自行绕过安全校验或禁用钩子。
- **一人一份上下文**：所有代码修改必须让未来的维护者能直接看懂，避免依赖口头约定或隐式状态。

## 2. 项目与技术栈

| 层级 | 技术 |
| --- | --- |
| 前端 | Taro 4 + React 18 + TypeScript + CSS Modules |
| 后端 | Spring Boot 3.2 + MyBatis + JDK 17 |
| 数据库 | MySQL 8.0（字符集 utf8mb4） |
| 消息队列 | RabbitMQ |
| 构建工具 | Maven 3.9.4 |
| 容器化 | Docker Compose（可选，本地以直接运行优先） |

本地开发地址：
- 前端 H5 预览：`http://localhost:10086/`
- 后端 API：`http://localhost:8080/`

## 3. 编码规范

### 3.1 前端

- 使用 TypeScript，禁止隐式 `any`；新组件优先使用函数式组件。
- 样式统一使用 CSS Modules，变量统一引用 `@/styles/variables.scss`。
- 组件职责单一，避免一个文件同时处理 UI、业务逻辑、网络请求；状态管理使用 Zustand。
- API 请求统一封装到 `src/services/api.ts`，返回 Promise，错误处理在调用处完成。
- 不要写死地址；环境相关配置通过 `process.env.TARO_ENV` 区分：
  - H5 开发走 `http://localhost:8080`
  - 微信小程序发布走远程服务器 `http://47.92.70.56:8080`
- 不允许使用占位图片；所有图片资源必须通过项目指定接口生成或用户明确提供。

### 3.2 后端

- 所有实体类使用 Lombok 减少样板代码，但必须显式定义字段类型与约束。
- MyBatis Mapper 优先使用注解；复杂 SQL 才使用 XML。
- Controller 层统一返回 `ApiResult<T>`，异常统一由全局异常处理捕获。
- 数据库操作必须在事务边界内完成；涉及多表写入（如题目+选项）使用 `@Transactional`。
- 接口路径统一前缀 `/api/`，RESTful 风格：
  - 查询：`GET /api/xxx`
  - 创建：`POST /api/xxx`
  - 更新：`PUT /api/xxx/{id}`
  - 删除：`DELETE /api/xxx/{id}`

### 3.3 数据库

- 所有表必须使用 `CHARSET=utf8mb4`，避免中文乱码。
- JDBC URL 必须包含：
  ```
  useUnicode=true&characterEncoding=UTF-8&serverTimezone=Asia/Shanghai&connectionCollation=utf8mb4_unicode_ci
  ```
- 枚举值在 Java 中使用 Enum，在数据库中存储为字符串或数值，禁止用魔法数字。
- 新增字段必须评估是否需要索引；外键字段建议建索引。

## 4. 安全标准

- **防注入**：禁止字符串拼接 SQL；所有参数使用 `#{}` 占位符。
- **防 XSS**：用户输入内容在展示前必须转义；禁止将用户输入直接渲染为 HTML。
- **防越权**：教师端接口后续必须接入身份校验；当前阶段任何管理操作禁止默认允许匿名调用。
- **敏感信息**：密码、密钥、AppID、上传密钥等禁止硬编码到仓库；使用 `application-dev.yml` 或环境变量。
- **文件上传**：图片/音频先走 Nginx 静态资源；上传接口必须校验文件类型、大小，禁止可执行文件。
- **日志脱敏**：日志中不得打印用户密码、Token、手机号等敏感字段。

## 5. 最佳实践

- **先读后改**：修改任何文件前必须先用 Read 工具查看内容，禁止基于猜测修改。
- **小步提交**：单个功能一个 commit，避免一次性改动过多文件；commit 信息聚焦「为什么」而非「做了什么」。
- **代码复用优先**：同类组件、工具函数优先复用已有实现，不重复造轮子。
- **依赖审慎**：新增依赖必须评估必要性，优先使用项目已有技术栈内的方案。
- **文档与注释**：注释只写「为什么」，不写「做什么」；禁止提交临时调试代码（如 `console.log`、`debugger`）。

## 6. 伦理准则

- 不生成、不传播、不协助生成违法违规内容。
- 不收集超出功能必需范围的用户个人信息。
- 教师端题目管理、学生答题数据属于教育场景敏感数据，必须保证数据完整性与可追溯性。
- 不得为规避审核、伪造数据、刷量、作弊等目的编写代码。
- 当用户请求可能违反安全或伦理规范时，明确拒绝并说明原因。

## 7. 错误处理流程

1. **复现**：先在本地或测试环境稳定复现问题，禁止直接修改线上环境。
2. **定位**：通过日志、断点、最小化测试用例定位根因，不凭经验猜测。
3. **修复**：只修改最小必要代码；修复后检查是否引入回归。
4. **验证**：本地运行并通过实际场景验证；后端接口使用 PowerShell/curl 测试。
5. **记录**：在提交说明中简述问题根因与修复方案；必要时补充单元测试。

## 8. 版本控制要求

- 使用 Git 管理代码；每次开发前确认当前分支状态。
- **禁止**以下未授权操作：
  - `git push --force`
  - `git reset --hard`
  - `git checkout .`
  - 删除他人分支或标签
  - 跳过 Git 钩子（`--no-verify`）
- 仅在用户明确要求时创建 commit；提交前检查是否有敏感文件（`.env`、密钥等）。
- Commit message 使用中文，格式：`<类型>: <简短描述>`，例如：
  - `feat: 新增教师端题目删除接口`
  - `fix: 修复 MySQL 中文乱码`
  - `refactor: 抽离题目选项渲染组件`

## 9. 团队协作规范

- **任务跟踪**：复杂任务使用 TODO 列表拆分，完成一项标记一项。
- **信息同步**：修改涉及多个文件或影响既有行为时，在回复中清晰列出改动点与验证结果。
- **代码审查**：所有功能改动完成后，必须进行自测；关键逻辑建议第二人复核。
- **沟通语言**：与用户沟通使用中文；代码注释、提交信息同样使用中文，除非项目另有约定。
- **尊重用户决策**：技术选型、产品设计以用户最终决定为准；提供建议但不替用户做决定。

## 10. 开发前检查清单

每次开始工作前，请确认：

- [ ] 已阅读本规范。
- [ ] 已了解当前任务范围，不超出用户要求。
- [ ] 已查看相关文件，知道要改哪里。
- [ ] 本地服务（后端 `8080`、前端 `10086`）可正常启动或已确认不需要启动。
- [ ] 涉及数据库变更时，已备份或确认 schema.sql / data.sql 可重建。
- [ ] 完成后将进行本地验证，并给出验证截图或测试结果。
